Angular 编码风格指南

edit

正在寻找关于 Angular 语法、约定和应用结构的权威指南？ 请看这里。 本风格指南介绍了推荐的约定，更重要的是，解释了原因。

风格词汇表

每条准则都描述了一个好的或坏的实践，并且都具有一致的呈现方式。

每条准则的措辞都表明了建议的强度。

Do 是指应该始终遵循的准则。 Always 可能有点太绝对了。 字面上应该始终遵循的准则非常罕见。 另一方面，你需要一个非常特殊的情况才能打破 Do 准则。

Consider 准则通常应该遵循。 如果你完全理解准则背后的含义，并且有充分的理由偏离，那么就这样做。 目标是保持一致。

Avoid 表示你应该几乎从不做的某些事情。 avoid 的代码范例具有明显的红色标头。

Why？
给出遵循先前建议的原因。

文件结构约定

一些代码范例显示了一个文件，该文件具有一个或多个类似命名的配套文件。 例如，hero.component.ts 和 hero.component.html。

该指南使用快捷方式 hero.component.ts|html|css|spec 来表示这些不同的文件。 使用此快捷方式使本指南的文件结构更易于阅读且更简洁。

单一职责

将 单一职责原则 (SRP) 应用于所有组件、服务和其他符号。 这有助于使应用更清晰、更易于阅读和维护，以及更易于测试。

单一规则

风格 01-01

Do 每个文件定义一个事物，例如服务或组件。

Consider 将文件限制为 400 行代码。

Why？
每个文件一个组件使代码更易于阅读、维护，并避免与源代码控制中的团队发生冲突。

Why？
每个文件一个组件可以避免隐藏的错误，这些错误通常在将组件组合在一个文件中时出现，在这些文件中，组件可能会共享变量、创建不必要的闭包或与依赖项产生不必要的耦合。

Why？
单个组件可以是其文件的默认导出，这有助于通过路由器进行惰性加载。

关键是使代码更可复用、更易于阅读且不易出错。

命名

命名约定对于可维护性和可读性至关重要。 本指南推荐了文件名和符号名称的命名约定。

通用命名准则

风格 02-01

Do 对所有符号使用一致的名称。

Do 遵循一种模式，该模式描述符号的特性，然后是其类型。 推荐的模式是 feature.type.ts。

Why？
命名约定有助于提供一种一致的方式来快速查找内容。 项目内的一致性至关重要。 与团队保持一致性很重要。 整个公司的一致性提供了巨大的效率。

Why？
命名约定应有助于更快地找到所需的代码，并使其更易于理解。

Why？
文件夹和文件的名称应清楚地表达其意图。 例如，app/heroes/hero-list.component.ts 可能包含一个管理英雄列表的组件。

用点和破折号分隔文件名

风格 02-02

Do 使用破折号分隔描述性名称中的单词。

Do 使用点分隔描述性名称和类型。

Do 对所有组件使用一致的类型名称，遵循一种模式，该模式描述组件的特性，然后是其类型。 推荐的模式是 feature.type.ts。

Do 使用传统的类型名称，包括 .service、.component、.pipe、.module 和 .directive。 如果必须，可以创建其他类型名称，但要注意不要创建太多。

Why？
类型名称提供了一种一致的方式来快速识别文件中的内容。

Why？
类型名称使使用编辑器或 IDE 的模糊搜索技术轻松找到特定文件类型。

Why？
未缩写的类型名称（如 .service）具有描述性且明确。 诸如 .srv、.svc 和 .serv 之类的缩写可能会造成混淆。

Why？
类型名称为任何自动化任务提供模式匹配。

符号和文件名

风格 02-03

Do 对所有资产使用一致的名称，这些名称以它们所代表的内容命名。

Do 对类名使用大驼峰命名法。

Do 将符号的名称与文件的名称匹配。

Do 为该类型的事物在符号名称后附加常规后缀（例如 Component、Directive、Module、Pipe 或 Service）。

要 为该类型的文件名添加约定俗成的后缀（例如 .component.ts、.directive.ts、.module.ts、.pipe.ts 或 .service.ts）。

为什么？
一致的约定可以轻松快速地识别和引用不同类型的资产。

服务名

风格 02-04

要 对所有以特性命名的服务使用一致的名称。

要 在服务类名后添加 Service 后缀。 例如，用于获取数据或英雄的服务应该称为 DataService 或 HeroService。

一些术语明确地表示服务。 它们通常以 "-er" 结尾来表示执行者。 你可能更喜欢将记录消息的服务命名为 Logger 而不是 LoggerService。 请确定你的项目中是否可以接受此例外。 与往常一样，力求保持一致。

为什么？
提供了一种一致的方式来快速识别和引用服务。

为什么？
像 Logger 这样清晰的服务名称不需要后缀。

为什么？
像 Credit 这样的服务名称是名词，需要后缀，并且当不清楚它是服务还是其他东西时，应该使用后缀命名。

指令选择器

风格 02-06

应该 对指令的选择器命名使用小驼峰命名法。

为什么？

为什么？
Angular HTML 解析器是区分大小写的，并且可以识别小驼峰命名法。

为什么？
防止名称冲突。

为什么？
指令很容易识别。

应用结构和 NgModules

对实现有一个近期规划，并有一个长期愿景。 从小处着手，但要牢记应用的发展方向。

应用的所有代码都放在名为 src 的文件夹中。 所有特性区域都在它们自己的文件夹中。

所有内容都是每个文件一个资产。 每个组件、服务和管道都在其自己的文件中。 所有第三方供应商脚本都存储在另一个文件夹中，而不是 src 文件夹中。 在本指南中使用文件的命名约定。

总体结构指南

风格 04-06

应该 从小处着手，但要牢记应用未来的发展方向。

应该 对实现有一个近期规划，并有一个长期愿景。

应该 将应用的所有代码都放在名为 src 的文件夹中。

可以考虑 在组件有多个配套文件（.ts、.html、.css 和 .spec）时，为其创建一个文件夹。

为什么？
有助于在早期阶段保持应用结构小巧且易于维护，同时在应用增长时易于演变。

为什么？
组件通常有四个文件（例如，*.html、*.css、*.ts 和 *.spec.ts），并且会很快使文件夹变得杂乱。

按特性分文件夹 结构

风格 04-07

应该 创建以其代表的特性区域命名的文件夹。

为什么？
开发人员可以快速定位代码并识别每个文件代表的内容。 该结构尽可能扁平，并且没有重复或冗余的名称。

为什么？
通过组织内容，帮助减少应用变得杂乱。

为什么？
当文件很多时，例如 10 个以上，使用一致的文件夹结构可以更容易地定位它们，而在扁平结构中则更困难。

特性模块

样式 04-09

要为应用中所有不同的特性创建一个 NgModule；例如，Heroes 特性。

要将特性模块放在与特性区域同名的文件夹中；例如，在 app/heroes 中。

要命名特性模块的符号，使其反映特性区域、文件夹和文件的名称；例如，app/heroes/heroes.module.ts 定义了 HeroesModule。

为什么？
特性模块可以向其他模块公开或隐藏其实现。

为什么？
特性模块可以很容易地被急性加载和惰性加载路由。

为什么？

为什么？
特性模块有助于明确责任，并使不同团队更容易分配开发职责。

为什么？
特性模块可以很容易地被隔离以进行测试。

共享特性模块

样式 04-10

要在 shared 文件夹中创建一个名为 SharedModule 的特性模块；例如，app/shared/shared.module.ts 定义了 SharedModule。

要在一个共享模块中声明组件、指令和管道，当这些条目将被其他特性模块中声明的组件复用和引用时。

考虑当共享模块的内容在整个应用中被引用时，使用名称 SharedModule。

考虑不要在共享模块中提供服务。 服务通常是单例，它们为整个应用或特定的特性模块提供一次。 但是，也有例外情况。 例如，在下面的示例代码中，请注意 SharedModule 提供了 FilterTextService。 这在这里是可以接受的，因为该服务是无状态的；也就是说，服务的消费者不受新实例的影响。

要导入 SharedModule 中的资产所需的所有模块；例如，CommonModule 和 FormsModule。

为什么？
SharedModule 将包含组件、指令和管道，它们可能需要来自另一个通用模块的特性；例如，CommonModule 中的 ngFor。

要在 SharedModule 中声明所有组件、指令和管道。

要从 SharedModule 导出其他特性模块需要使用的所有符号。

为什么？
SharedModule 的存在是为了使常用的组件、指令和管道可用于许多其他模块中组件的模板。

避免在 SharedModule 中指定应用范围内的单例提供者。 有意的单例是可以的。 请注意。

为什么？
惰性加载的特性模块如果导入了该共享模块，将创建该服务自己的副本，并可能产生不良结果。

为什么？
你不希望每个模块都有其自己单独的单例服务实例。 然而，如果 SharedModule 提供服务，则确实存在发生这种情况的风险。

将模板和样式提取到它们自己的文件中

样式 05-04

要将模板和样式提取到单独的文件中，当超过 3 行时。

务必将模板文件命名为 [component-name].component.html，其中 [component-name] 是组件名称。

务必将样式文件命名为 [component-name].component.css，其中 [component-name] 是组件名称。

为什么？
当您移动组件文件时，组件相对 URL 不需要更改，只要这些文件保持在一起。

为什么？
./ 前缀是相对 URL 的标准语法；不要依赖 Angular 当前在没有该前缀的情况下工作的能力。

使用 @Injectable() 类装饰器

风格 07-04

务必在使用类型作为服务的依赖的令牌时，使用 @Injectable() 类装饰器而不是 @Inject 参数装饰器。

为什么？
Angular 依赖注入（DI）机制基于服务构造函数参数的声明类型来求解服务自身的依赖。

为什么？
当服务仅接受与类型令牌关联的依赖项时，与在每个单独的构造函数参数上使用 @Inject() 相比，@Injectable() 语法要简洁得多。

app/heroes/shared/hero-arena.service.ts

      
import {Inject} from '@angular/core';import {HttpClient} from '@angular/common/http';import {HeroService} from './hero.service';/* avoid */export class HeroArena {  constructor(    @Inject(HeroService) private heroService: HeroService,    @Inject(HttpClient) private http: HttpClient,  ) {}}

app/heroes/shared/hero-arena.service.ts

      
import {Injectable} from '@angular/core';import {HttpClient} from '@angular/common/http';import {HeroService} from './hero.service';@Injectable()export class HeroArena {  constructor(    private heroService: HeroService,    private http: HttpClient,  ) {}  // test harness  getParticipants() {    return this.heroService.getHeroes();  }}

数据服务

通过服务与服务器通信

风格 08-01

务必将用于执行数据操作和与数据交互的逻辑重构到服务中。

务必使数据服务负责 XHR 调用、本地存储、内存存储或任何其他数据操作。

为什么？
组件的职责是呈现和收集视图的信息。 它不应该关心如何获取数据，而只需知道向谁请求数据即可。 分离数据服务会将关于如何获取数据的逻辑转移到数据服务，并使组件更简单，更专注于视图。

为什么？
在测试使用数据服务的组件时，这使得更容易测试（模拟或真实）数据调用。

为什么？
数据管理的细节（例如标头、HTTP 方法、缓存、错误处理和重试逻辑）与组件和其他数据使用者无关。

数据服务封装了这些细节。 在服务内部演化这些细节更容易，而不会影响其使用者。 并且使用模拟服务实现来测试使用者也更容易。
